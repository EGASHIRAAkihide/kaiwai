# ビジネス要件定義書 (BRD): KAIWAI

## 1. プロジェクト概要

* **製品名:** KAIWAI（界隈）
* **コンセプト:** 「場所」に紐づく熱量を可視化する、ローカル・コミュニティ・ハブ。
* **ミッション:** 共通の拠点（ジム、公園、コース）を持つトレーニーを繋げ、個人のトレーニングを競技的かつ社交的な体験へとアップグレードする。

## 2. ターゲットユーザー

1. **コア層:** HYROXやマラソン大会に出場する、目的意識の高いアスリート。
2. **リーダー層:** 独自の練習メソッドを持つインフルエンサー、プロダクトマネージャー、トレーナー。
3. **ローカル層:** 祐天寺や皇居など、特定のエリアをホームグラウンドとする市民ランナー。

## 3. 主要機能要件 (Feature Requirements)

| カテゴリ | 機能名 | 詳細・ビジネス価値 |
| --- | --- | --- |
| **スポット・ベース** | **界隈（スポット）登録** | `flutter_map`を活用し、特定の練習場所を「界隈」として定義。APIキー不要のOSM運用でコストを抑える。 |
| **リアルタイム性** | **チェックイン・トラッカー** | `geolocator`による位置情報判定。特定の「界隈」にユーザーが流入した際、滞在人数をカウントしライブ感を演出。 |
| **ソーシャル/競技** | **界隈別ランキング** | そのスポット限定のタイムや走行距離のリーダーボード。同じ場所を走るライバルを可視化。 |
| **CMS/収益化** | **界隈ノート（Paywall）** | TiptapベースのCMSで、リーダーがそのスポット向けの練習メニューを配信。一部を有料化し、コミュニティ運営を収益化。 |
| **技術拡張** | **モーション解析連携** | （将来）ChoreoIDの技術を転用し、スクワットやランジのフォームをカメラで解析・評価する機能を界隈ごとに提供。 |

## 4. 非機能要件 (Technical Context)

* **アーキテクチャ:**
* **Frontend:** Flutter (iOS/Android) - `flutter_map`による自由度の高い地図UI。
* **Backend:** Next.js (Admin/Web) + Supabase (PostgreSQL/PostGISによる位置情報計算)。


* **プライバシー:**
* ユーザーの正確な位置情報は共有せず、「どの界隈（エリア）にいるか」のみを匿名的に表示するモードを搭載。


* **コスト効率:**
* Google Maps APIを避け、OSM（OpenStreetMap）を採用することで、MAU増加時のランニングコストを最小化。



## 5. ユーザー体験（User Story）

1. **発見:** アプリを開くと、自分の周囲にある「活発な界隈（例：駒沢公園ランナー界隈）」が地図上に光って見える。
2. **参加:** その界隈にチェックインすると、その日の「界隈ノート（推奨メニュー）」が届く。
3. **競争:** 走り終わると、同じ界隈のメンバー内での今日の順位が表示される。
4. **継続:** 有名なリーダーが有料で配信する「HYROX対策・特別界隈ノート」を購読し、より高度な練習に取り組む。

## 6. 成功指標 (KPI)

* **界隈定着率:** 特定のスポットに週3回以上チェックインするユーザーの割合。
* **ノート購読数:** リーダーが発信する有料コンテンツの決済コンバージョン率。
* **滞在時間:** アプリ内で他のユーザーの活動（ランキングや投稿）を確認している時間。
